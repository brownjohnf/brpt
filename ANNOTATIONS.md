# Annotations

Annotations attach contextual notes to specific lines of a
document or diff. They appear as highlighted blocks interleaved
with the rendered content at the exact positions they reference,
and are marked in the gutter with blue dots.

Annotations are persisted to disk in brpt's sidecar storage
(`~/.brpt/sidecars/`) and survive tab close and app restart.
Each annotation has a unique ID and can be dismissed from the
UI or removed via the CLI.

## Quick start

```bash
# Add an annotation at line 12
echo "This has a bug" | brpt annotate add document.md 12 -

# Add an annotation spanning lines 10–15
brpt annotate add document.md 10:15 "Refactor this block"

# Bulk import from a JSON file
brpt annotate import document.md annotations.json
```

The annotation renders as a styled block at the target line,
with a blue dot in the gutter. Click the × button on an
annotation to dismiss it.

## CLI reference

Annotations are managed through the `brpt annotate`
subcommand:

### `annotate add <target> <line> <message>`

Add a single annotation. `<line>` is a line number or a range
(`start:end`). The message is markdown.

```bash
# Single line
brpt annotate add src/main.py 42 "Use **const** here"

# Line range
brpt annotate add src/main.py 10:20 "Extract into function"

# Read message from stdin
echo "Long note with \`code\`" | brpt annotate add file.md 5 -

# Read message from a file
brpt annotate add file.md 5 ignored --message-file note.md

# Tag with a source identifier
brpt annotate add file.md 5 "Issue" --source myLinter
```

### `annotate import <target> <annotations-file>`

Bulk import annotations from a JSON file. The file must contain
an `annotations` array:

```json
{
  "annotations": [
    {
      "line": 5,
      "format": "markdown",
      "content": "Consider using `const` here."
    },
    {
      "startLine": 10,
      "endLine": 15,
      "format": "markdown",
      "content": "This block needs an index on `created_at`."
    }
  ]
}
```

The import normalizes the input: a `line` field is converted to
matching `startLine`/`endLine` values. Each imported annotation
gets a unique ID.

```bash
brpt annotate import document.md review-comments.json
```

### `annotate remove <target> <annotation-id>`

Remove a specific annotation by its ID.

```bash
brpt annotate remove document.md "2025-03-04T12:00:00.000Z-0"
```

### `annotate clear <target>`

Remove all annotations for a file.

```bash
brpt annotate clear document.md
```

### `annotate list <target>`

List all annotations for a file as JSON. Useful for scripting
or inspecting current state.

```bash
brpt annotate list document.md
```

## Annotation fields

Each annotation stored in the sidecar has:

| Field | Description |
|-------|-------------|
| `id` | Unique identifier (auto-generated) |
| `startLine` | Start of the target range (1-indexed) |
| `endLine` | End of the target range (1-indexed; same as `startLine` for single-line) |
| `format` | Content format — currently `"markdown"` |
| `content` | The annotation body, rendered as markdown |
| `source` | Optional source identifier (e.g. tool name, agent name) |

When importing, you can use either `line` (single line) or
`startLine`/`endLine` (range). The `format` field defaults to
`"markdown"` if omitted.

## Works with all viewing modes

### Plain markdown files

```bash
brpt document.md
brpt annotate add document.md 5 "Note about line 5"
```

Line numbers refer to the source file's lines.

### Diffs from a unified diff file

```bash
brpt diff document.md changes.patch
brpt annotate add document.md 12 "Check this change"
```

Line numbers refer to lines in the diff file itself.

### Diffs computed from two files

```bash
brpt diff-files new.md old.md
brpt annotate add new.md 8 "Why was this removed?"
```

Line numbers refer to the **new** file's source lines. brpt
maps these to the corresponding positions in the rendered diff
automatically.

## Gutter interaction

- **Blue dots** appear next to annotated lines. Click a dot to
  collapse/expand the annotations at that position.
- **Range lines** connect the start and end dots of a range
  annotation with a vertical bar.
- **Dismiss** — click the × on an annotation block to remove it.
- Line numbers are displayed in the gutter alongside the dots.

## Storage

Annotations are stored in brpt's sidecar directory
(`~/.brpt/sidecars/`), keyed by a hash of the target file
path. They persist across app restarts and tab close/reopen.
The sidecar also stores notifications for the same file.

This is different from the import file: importing reads
annotations from a JSON file and copies them into the sidecar.
The original JSON file is not watched or referenced afterward.

## Intended use cases

Annotations are designed to be **generated by other tools**, not
written by hand (though you can). The CLI is simple enough that
any script can add annotations.

### Code review

A review tool examines a file, produces line-specific comments,
and adds them as annotations:

```bash
# Your review tool adds annotations directly
my-review-tool document.md | while IFS=: read line msg; do
  brpt annotate add document.md "$line" "$msg"
done
```

Or bulk import from a structured output:

```bash
my-review-tool document.md --json > review.json
brpt annotate import document.md review.json
```

### AI agent feedback

An AI agent analyzing code can add annotations as it works.
The preview updates live as annotations arrive:

```bash
brpt annotate add src/main.py 42 "Bug: off-by-one" \
  --source claude
brpt annotate add src/main.py 100:110 "Extract method" \
  --source claude
```

The `--source` flag tags annotations with the agent's identity.

### Linter/static analysis overlay

Transform linter output into annotations:

```bash
eslint src/app.ts --format json \
  | my-eslint-to-annotations > lint.json
brpt annotate import src/app.ts lint.json
```

### Diff review with inline commentary

Review a PR by annotating the diff:

```bash
brpt diff src/feature.ts changes.patch
brpt annotate add src/feature.ts 15 "Why this change?"
```

### Teaching and documentation

Annotate a code example with explanations. Each annotation body
is full markdown, so you can include formatted text, code
blocks, links, and lists to explain what each section does.

### Stdin for piping

Commands that produce markdown output can pipe directly into
annotations:

```bash
cat <<'EOF' | brpt annotate add doc.md 5 -
This is a **multi-line** annotation.

```python
def fixed():
    return True
```
EOF
```
